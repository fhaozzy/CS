# PPT 讲稿（20分钟版）

建议总时长：20分钟

## Slide 1 封面（建议 1分钟）

大家好，今天我汇报的题目是《以 NVIDIA A100 为例理解 GPU 的体系结构与编程方式》。我想借 A100 这个比较有代表性的例子，回答三个问题：第一，GPU 和 CPU 的体系结构差别到底在哪里；第二，A100 这样的 GPU 内部是怎样组织起来的；第三，为什么这种硬件结构会直接改变程序员的写法。整场汇报我会先讲 CPU 和 GPU 的总体差异，再讲 A100 的总体结构、编程模型和内存层次，最后讲它适合做什么、不适合做什么，以及我自己的理解。

## Slide 2 CPU 与 GPU 的体系结构差异（建议 2分30秒）

这一页先从最根本的地方讲 CPU 和 GPU 的差别。CPU 的设计目标是低延迟和强通用性，所以它会把很多晶体管资源用在复杂控制逻辑、分支预测、乱序执行和多级缓存上。换句话说，CPU 要求的是面对复杂任务时，单个线程也能尽快跑完。GPU 的目标不一样。GPU 更强调高吞吐，所以它把更多资源放在大量并行计算单元上，而不是把每一个核心都做得特别复杂。它更像是把很多相对简单但数量巨大的计算单元组织起来，同时处理一大批结构相似的数据。所以可以把 CPU 理解成擅长快速处理少量复杂任务，把 GPU 理解成擅长同时处理海量相似任务。两者不是先进和落后的关系，而是面向不同工作负载形成的两种体系结构路线。

## Slide 3 为什么选择 A100（建议 1分30秒）

我选择 A100 作为案例，是因为它非常典型。A100 是数据中心场景里非常有代表性的一代 GPU，它不是单纯追求图形渲染，而是明显面向 AI 训练、推理和高性能计算。在这块芯片上，我们可以同时看到几个现代 GPU 的关键特征：大规模的 SM 阵列、高带宽的 HBM2e、针对矩阵运算优化的 Tensor Core，以及和硬件深度绑定的 CUDA 软件生态。所以用 A100 来讲 GPU，有一个好处，就是既能讲清楚硬件结构，也能顺着硬件结构讲到编程方式，最后再联系到 AI 和 HPC 为什么离不开这种架构。

## Slide 4 A100 总体结构图（建议 2分30秒）

这一页看 A100 的总体结构。最左边是 Host CPU，它的职责主要是控制程序流程，比如准备数据、分配内存、启动 kernel，以及决定什么时候让 GPU 开始执行。中间是 A100 GPU 芯片本体。这里最重要的一点是，它不是一个像 CPU 那样的大核心，而是很多个 SM，也就是 Streaming Multiprocessor，组成的大规模并行阵列。每个 SM 内部又有自己的计算单元、寄存器和共享资源。最右边是 HBM 高带宽显存。GPU 要想持续保持高吞吐，光有很多计算单元还不够，还必须持续喂给它们足够的数据，所以高带宽存储在整个结构里非常关键。如果把这一页压缩成一句话，就是 CPU 负责组织和调度，GPU 负责大规模并行执行，HBM 负责持续供给数据，而中间的互连和缓存负责把这些部分连成一个整体。

## Slide 5 程序员视角和硬件视角如何对上号（建议 2分30秒）

这一页想回答一个很关键的问题，就是程序员写出来的并行结构，最后到底是怎么落到硬件上的。从编程模型看，我们平时写的是 Grid、Block 和 Thread。Grid 可以理解成一次 kernel 启动后形成的任务全集，Grid 下面再分成很多 Block，每个 Block 里面又有很多 Thread。但从硬件角度看，真正执行这些任务的是 GPU 上的很多个 SM，而线程并不是一个一个独立调度的，硬件通常以 Warp 为基本执行单位。对于 NVIDIA 来说，一个 Warp 通常是 32 个线程。这件事非常重要，因为它直接影响程序性能。比如一个 Block 会被分配到某个 SM 上执行，Warp 内线程如果走不同分支，就会出现分支发散，效率下降。所以 GPU 编程不是简单把任务拆成很多线程就结束了，程序员还要主动考虑线程块大小、线程访问数据的方式，以及如何尽量让同一个 Warp 里的线程做相似的事情。

## Slide 6 CPU / GPU 代码对比（建议 3分钟）

这一页我想用最直观的方式比较 CPU 和 GPU 的编程差异。左边是标准 C++ 版本的向量相加，核心就是一个 for 循环。它的思路非常自然：把算法写清楚，然后让编译器和 CPU 去做流水线、缓存和指令级并行方面的优化。右边是 CUDA C++ 版本。这里首先要显式写一个 kernel，然后每个线程要通过 blockIdx、blockDim 和 threadIdx 算出自己负责处理的数据下标。也就是说，在 GPU 上，程序员不能只写算法，还必须把算法映射到线程组织上。下面这一行主机端流程也很关键。GPU 版本通常还要额外做几件事：先在 GPU 显存里申请空间，再把数据从主机内存拷到显存里，然后启动 kernel，最后把结果再拷回来。这说明 GPU 编程和 CPU 编程有一个根本差异：CPU 编程更像是在描述顺序计算过程，而 GPU 编程不仅要描述计算，还要显式管理并行结构和数据移动。顺便说一句，像向量相加这种例子本身并不复杂，真正的价值不是它算得多快，而是它很清楚地暴露出了 CPU 和 GPU 编程思维的不同。

## Slide 7 内存层次与延迟隐藏（建议 2分30秒）

这一页解释 GPU 为什么能把访存延迟隐藏起来。左边是一个简化的内存层次图，越往上越快，但容量越小；越往下容量越大，但延迟也越高。寄存器最快，shared memory 也很快，再往下是 L2 和 HBM。如果一个线程去访问比较慢的显存，它本身会等待。但是 GPU 的做法不是让这个线程一直占着执行资源，而是当某个 Warp 因为访存停住时，调度器立刻切换到另一个已经准备好的 Warp。所以 GPU 隐藏延迟的核心思路，不是把单个线程做得特别强，而是同时准备很多线程上下文，用线程级并发去覆盖访存等待。这也解释了为什么 GPU 优化里经常会提到访问合并、shared memory、occupancy 这些概念。因为一旦数据访问不规整，或者同时可运行的 Warp 不够多，延迟就很难被真正隐藏掉。

## Slide 8 A100 擅长什么，不擅长什么（建议 2分钟）

这一页想说明，A100 很强，但它绝对不是所有任务都擅长。它最适合的是并行度很高、数据访问相对规则、不同线程做的事情比较相似的任务。比如矩阵乘法、卷积、批量图像处理和大规模张量运算，这些都很适合放到 GPU 上。但如果任务本身很不规则，比如大量链表、树结构、复杂事务逻辑，或者线程之间分支差异很大，那么 GPU 的优势通常就不明显，甚至可能还不如 CPU。所以判断一个程序该不该上 GPU，关键不只是看计算量大不大，而是看它有没有足够的并行性，访存是不是规整，控制流是不是足够统一。

## Slide 9 A100 的关键特征与个人判断（建议 1分30秒）

这一页我把 A100 最值得关注的几个特征收一下。SM 阵列决定了并行规模，HBM2e 决定了持续供数能力，Tensor Core 决定了矩阵和张量类任务的高吞吐，而 CUDA 生态则让程序员能够稳定地把软件映射到这套硬件上。我个人的判断是，理解 A100 不能只盯着参数表看多少 TFLOPS、多少显存，更重要的是看这些部件是不是形成了一个相互匹配的体系。从体系结构角度看，A100 的代表性就在于它不是单点很强，而是计算、存储、调度和软件模型一起围绕高吞吐做了协同设计。

## Slide 10 总结（建议 1分钟）

最后总结一下。第一，GPU 和 CPU 的根本区别，不是谁更先进，而是高吞吐和低延迟两种不同设计目标的区别。第二，A100 的代表性在于它把并行结构、高带宽存储和软件编程模型做成了一整套方案。第三，GPU 之所以会改变程序员的写法，是因为它的执行单位、调度方式和数据流动路径都和 CPU 不一样。所以我对 A100 的理解，不是把 CPU 简单做得更快，而是为了吞吐量重新分配资源的一种处理器设计。我的汇报到这里结束，谢谢大家。
